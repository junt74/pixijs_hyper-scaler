# 技術的課題整理：真3D（Doc1方式）Super Scaler Engine

> アーキテクチャ確定：疑似3Dを廃棄し、Doc1方式（cannon-es + PixiJS投影）のみを採用。

---

## 🔴 最優先・設計フェーズで解決必須

### 1. cannon-esとPixiJSの同期戦略
物理演算のステップ（固定Δt）とレンダリングフレーム（可変fps）は非同期で動く。単純に毎フレーム座標をコピーするだけでは高速移動時にカクつきが出る。
**Fixed Timestep + Linear Interpolation（補間）** の実装が必要。

#### 方針（確定）：手動accumulator管理による補間

- レンダリングの固定フレーム化は採用しない。rAFはデバイスのリフレッシュレートに依存するため制御不可能であり、処理落ち時に再び非同期になる根本的な脆さがある。
- cannon-esの `world.step()` 内部サブステップモードは使用しない（補間係数αが取得できないため）。
- 実装方針：毎フレーム `accumulator += dt` で時間を積算し、固定ステップ分だけ物理を消費。ステップ前に全Bodyの位置・Quaternionをコピー保存し、残余時間から `alpha = accumulator / FIXED_STEP` を算出してスプライト座標を線形補間（位置はlerp、回転はslerp）してから投影する。
- 実装量は約30〜50行。技術的ブロッカーなし。

### 2. カメラ自体の物理表現
Galaxy Forceスタイルの場合、カメラ（＝自機）も3D空間を動く物理ボディになる。自機の慣性、バンク（傾き）、コースに沿った移動をcannon-es上でどう表現するかの設計が未定。
- レールシューター的な自動前進か、完全自由飛行かで実装規模が大きく変わる。
- 方針の早期確定が必要。

#### 方針（確定）：非衝突の独立物理ボディ

- カメラ（自機）はcannon-esの `Body` として存在するが、他のいかなるエンティティとも衝突しない。
- cannon-esのコリジョンフィルタリング（`collisionFilterGroup` / `collisionFilterMask`）でカメラ専用グループを設定し、マスクを `0` にすることで完全に衝突判定から除外する。
- 位置・慣性・バンク角の計算は物理エンジンに委ねつつ、当たり判定はゲームロジック側（弾・敵との距離判定など）で別途処理する。

#### 方針（確定）：自動前進レールシューター

- カメラはレール（ウェイポイント列）上を自動前進する。プレイヤーは前進速度を操作できない。
- プレイヤーが操作できるのはレール基準のローカルXY軸方向のオフセットのみ（スクリーン上の左右・上下移動）。
- 逆走は想定しない。
- cannon-esへの対応：カメラBodyに対し、毎フレームウェイポイント方向への速度を直接設定する（物理的な「引き寄せ」ではなく速度代入）。プレイヤー入力分のXY速度を加算し、慣性はcannon-esのdamping設定で表現する。
- バンク（ROLL）は入力のX速度から算出し、`sprite.rotation` に適用する（項目6と同方針）。

### 3. 地形コライダーの粒度問題
Blenderから出力されたJSON座標に静的Bodyを並べる方式だと、座標の間隔が粗ければ「すり抜け」、細かすぎれば「物理ボディ数が膨大」になる。
- 適切な粒度の検証が必要。
- Heightfield / Trimesh コライダーの使用可否をcannon-esの仕様と照合する必要がある。

#### 方針（確定）：Blenderキューブ組み合わせによるBoxコライダー

- 地形コライダーはスプライトエンティティとは独立したBlenderオブジェクトとして作成する。
- コライダーの単位はキューブ（Box）とし、cannon-esの `Box` シェイプにそのまま対応させる。
- 各キューブは以下のパラメータを持ち、BlenderのCustom PropertiesからJSONへエクスポートする。
  - 位置：`X`, `Y`, `Z`
  - 回転：`YAW`, `PITCH`, `ROLL`（オイラー角、度またはラジアン）
  - スケール：`SCALE_X`, `SCALE_Y`, `SCALE_Z`（Boxの half-extents として使用）
- これにより粒度問題・すり抜け問題をBlender側の配置精度で制御でき、Trimesh / Heightfield の複雑な仕様照合は不要になる。

## 🟡 実装フェーズで判断が必要

### 4. Quaternionベースのカメラ変換コスト
全エンティティ × 毎フレームのWorld→Camera変換はCPUバウンドになりやすい。エンティティ数が数百を超える場合、以下の対策を検討する。
- 行列演算のバッチ化
- `Float32Array` による手動計算への切り替え

#### 方針（確定）：フラスタムカリングによる計算対象の削減

World→Camera変換を行う前に、カメラ空間での位置チェックで計算対象を絞り込む。

```text
localZ <= 0              → カメラ後方 → 変換・描画スキップ
localZ > DRAW_DIST       → 描画距離外 → 変換・描画スキップ
projX < -margin          → 画面左外   → 描画スキップ
projX > screenW + margin → 画面右外   → 描画スキップ
```

- `localZ` チェックはWorld→Camera変換の副産物として得られるため、追加コストはほぼゼロ。
- 投影後スクリーン座標チェックはFOV計算（内積・外積）不要で、Super Scalerの構造に最も適合する。
- 描画距離上限（`DRAW_DIST`）はObject Poolingの回収タイミングと統一する。
- エンティティ数が数百規模を超えた場合は、追加で `Float32Array` バッチ化を検討する。

### 5. Blenderエクスポーターのコライダー情報
見た目のメッシュ座標だけでなく、「このオブジェクトはBox / Sphere collider」という物理プリミティブ情報もBlenderから出力する必要がある。
- Blenderの **Custom Properties** を活用したJSONスキーマ設計が必要。

#### 方針（確定）：Collectionによる識別 + ネイティブTransformの直接出力

- `Colliders/` という名前のCollectionを作成し、コライダー用キューブをその中に配置する。
- CollectionはBlenderアウトライナー上のフォルダであり、エンティティ・オブジェクト・トランスフォームを持たない純粋な分類単位。
- 位置・回転・スケールはBlenderのオブジェクトTransformをそのまま読み取る。Custom Propertyへの手入力は不要。
- エクスポーターは `bpy.data.collections["Colliders"].objects` をイテレートして各キューブのTransformをJSONに出力する。

#### 注意：Blender ↔ cannon-es 座標系変換をエクスポーター側で定義すること

BlenderとcannoneーesはデフォルトのZ軸方向が異なるため、エクスポーター出力時に変換ルールを明確に定義し適用する。

| 軸 | Blender | cannon-es | 変換 |
| --- | --- | --- | --- |
| 左右 | X | X | そのまま |
| 上下 | Z（up） | Y（up） | Blender Z → cannon-es Y |
| 前後 | Y（forward） | Z | Blender Y → cannon-es Z（符号は確認要） |

- 変換ルールはエクスポータースクリプト内の1箇所に集約し、定数として定義する。
- 回転（YAW/PITCH/ROLL）のオイラー角も同様に軸の対応を明示する。
- 変換ルールが未定義のままコライダーを実装すると、全コライダーの向きと位置が一括でずれる致命的な問題につながる。

```json
{
  "colliders": [
    {
      "name": "Wall_01",
      "type": "box",
      "position":    { "x": 10.0, "y": 0.0,  "z": 5.0 },
      "rotation":    { "yaw": 0.0, "pitch": 45.0, "roll": 0.0 },
      "halfExtents": { "x": 2.0,  "y": 0.5,  "z": 3.0 }
    }
  ]
}
```

### 6. スプライトのバンクアニメーション
Galaxy Force等では自機が旋回するとスプライトが傾く。3D空間上のロール角をスプライトのフレーム番号にマッピングするロジックが必要。
- 物理ボディのQuaternion → ロール角（度） → アニメフレームインデックス の変換パイプラインを設計する。

#### 方針（確定）：カメラROLL角のビルボード回転への直接適用

- ROLLが発生するのはカメラ（自機）のみ。敵キャラ等の小型エンティティは旋回しても角度ではなく**状態**として管理するため、ROLL角の連続マッピングは不要。
- 対応するのは「カメラのROLL角 → ビルボードスプライトの回転」のみ。
- 実装方針：カメラBodyのQuaternionからROLL角を抽出し、PixiJSスプライトの `rotation` プロパティに直接適用する。フレームインデックスへの変換は行わない。

```text
camera.quaternion → rollAngle（ラジアン）→ sprite.rotation = rollAngle
```

### 7. コリジョングループ設計（カメラ以外のエンティティ）

cannon-esのコリジョンフィルタリングはビットマスク方式であり、Unityのレイヤー行列と同等の制御が可能。カメラ以外の各エンティティ間の衝突ルールを管理する仕組みを設計する必要がある。

#### 方針（確定）：レイヤー定数 ＋ 衝突マトリクス設定オブジェクトによる管理

- 各エンティティ種別をビットフラグ（2のべき乗）のレイヤー定数として定義する。
- 衝突ルール（どのレイヤー同士が衝突するか）は1箇所の衝突マトリクス設定オブジェクトに集約する。
- Bodyを生成する際、マトリクスから `collisionFilterMask` を自動計算して設定する。これによりBody生成コードには衝突ルールを記述せず、設定変更はマトリクスのみを修正すればよい。

レイヤー定義（暫定）：

| 定数名 | ビット | 備考 |
| --- | --- | --- |
| `LAYER_CAMERA` | `1 << 0` | mask = 0 確定（何とも衝突しない） |
| `LAYER_PLAYER` | `1 << 1` | 自機キャラクター（カメラとは別エンティティ） |
| `LAYER_ENEMY` | `1 << 2` | 敵キャラクター |
| `LAYER_PLAYER_BULLET` | `1 << 3` | 自弾 |
| `LAYER_ENEMY_BULLET` | `1 << 4` | 敵弾 |
| `LAYER_TERRAIN` | `1 << 5` | 静的地形コライダー |

- 各レイヤー間の衝突ON/OFFは実装フェーズで確定する。
- レイヤー追加（アイテム、ギミックなど）はビット定数を追加するだけで対応できる。

### 8. スプライトエンティティのBlenderエクスポート設計

ゲームワールドに配置する敵・装飾などのスプライトエンティティの初期配置情報をBlenderからエクスポートする設計が必要。

#### 方針（確定）：型別サブCollectionによる識別 + ネイティブTransformの直接出力

- `Sprites/` Collectionを頂点とし、エンティティ種別ごとにサブCollectionを作成する（例：`Sprites/EnemyA/`、`Sprites/Decoration_Tree/`）。
- 各エンティティはBlenderのEmpty（座標マーカー）として配置する。メッシュ形状は不要。
- エンティティ種別はサブCollection名から導出する。Custom Propertyへの手入力は不要。
- エクスポートするパラメータは位置（XYZ）とYAW回転のみ。スプライトはビルボード表示であるため、PITCH・ROLLは不要。
- 位置の座標変換ルールはコライダーエクスポーター（項目5）と共通とする。

```json
{
  "sprites": [
    {
      "type": "EnemyA",
      "position": { "x": 10.0, "y": 2.0, "z": 50.0 },
      "yaw": 0.0
    }
  ]
}
```

### 11. レールウェイポイントとトリガーボリュームの設計

カメラの前進経路（レール）と、ゲームイベントを発火するトリガーゾーンのBlender上での設計・エクスポート方針を定める。

#### 方針（確定）：Blender Curveによるレール設計 + 等間隔サンプリング出力

- レールはBlenderのCurveオブジェクト（Bezierカーブ）でコース中心線として設計する。
- エクスポーター側でカーブを等間隔にサンプリングし、XYZ点列としてJSONに出力する。
- ウェイポイントが持つデータは位置（XYZ）のみ。カメラの向きは前後ウェイポイントのベクトルから自動計算する。
- 前進速度の変化はトリガーボリュームで管理する（後述）。ウェイポイントが持つデータは位置（XYZ）のみで確定。

```json
{ "waypoints": [
    { "x": 0.0, "y": 0.0, "z":  0.0 },
    { "x": 5.0, "y": 0.5, "z": 50.0 }
  ]
}
```

#### 方針（確定）：Triggers/ CollectionとCustom Propertyによるトリガーボリューム

- `Triggers/` Collectionにキューブを配置し、各キューブの位置・回転・スケールをトリガーゾーンとして使用する。
- トリガー種別はキューブのCustom Property `type` で識別する。命名規則に応じてゲームロジック側の処理を分岐させる（例：`SpeedChange_120` → 前進速度を120に変更）。
- すべてのトリガーは**ワンショット**：カメラが進入した時点で1回だけ発火し、即座に破棄する。再発火はしない。
- 敵のスポーンはトリガーではなく、スプライトエンティティの個別判定（距離など）で処理する（項目8参照）。

```json
{ "triggers": [
    {
      "type": "Checkpoint",
      "position":   { "x": 0.0, "y": 0.0, "z": 100.0 },
      "rotation":   { "yaw": 0.0, "pitch": 0.0, "roll": 0.0 },
      "halfExtents": { "x": 10.0, "y": 5.0, "z": 2.0 }
    }
  ]
}
```

### 12. World→Screen 投影公式

Super Scalerの核心となる「3D World座標 → PixiJSスクリーン座標・スプライトスケール」の変換公式。すべてのパラメータを定数として明示し、都度調整できる形にする。

#### 方針（確定）：透視投影公式とチューニング定数の分離

チューニング定数（直接調整する値）：

| 定数名 | 初期値 | 説明 |
| --- | --- | --- |
| `SCREEN_W` | 640 | スクリーン横解像度（px） |
| `SCREEN_H` | 480 | スクリーン縦解像度（px） |
| `FOV_H` | 60 | 水平視野角（度） |
| `SPRITE_REF_DIST` | 100 | `scale = 1.0` になる基準距離（3Dユニット） |
| `DRAW_DIST` | 500 | 描画距離上限（項目4と共通） |

導出定数（`FOV_H` から自動計算、手動変更不要）：

```text
FOCAL = (SCREEN_W / 2) / tan(FOV_H * π / 180 / 2)
```

投影パイプライン：

```text
① World → Camera変換
   delta    = worldPos - camPos
   localPos = camQuat.conjugate() * delta

② フラスタムカリング（項目4と共通）
   localZ <= 0 または localZ > DRAW_DIST → スキップ

③ 透視投影（スクリーン座標）
   projX = (localX / localZ) * FOCAL + SCREEN_W / 2
   projY = (localY / localZ) * FOCAL + SCREEN_H / 2

④ スプライトスケール
   scale = SPRITE_REF_DIST / localZ
```

- `projX` / `projY` → `sprite.position.set(projX, projY)`
- `scale` → `sprite.scale.set(scale)`（`localZ = SPRITE_REF_DIST` のとき `scale = 1.0`）
- FOV感の調整は `FOV_H` のみを変更する。`FOCAL` は自動追従する。

### 13. スプライト深度ソート

カメラがYAW回転するコースでは、World座標のZでのソートは破綻する。カメラローカル空間の `localZ` でソートする必要がある。

#### 方針（確定）：localZによる毎フレームソート + PixiJS sortableChildren

- ソートキーは `localZ`（カメラローカル空間の奥行き）。World座標は使用しない。
- `localZ` は投影パイプライン①（World→Camera変換）の副産物として既に計算済みであり、追加コストはゼロ。
- フラスタムカリング後の描画対象のみソートするため、実際のN数は小さい。
- PixiJSへの適用：`container.sortableChildren = true` を設定し、各スプライトに `sprite.zIndex = -localZ` を毎フレーム代入する。PixiJSが描画直前に自動ソートする。

```text
localZ が小さい（近い） → -localZ が大きい → zIndex 大 → 前面に描画
localZ が大きい（遠い） → -localZ が小さい → zIndex 小 → 背面に描画
```

### 14. ステージJSONの構成設計
