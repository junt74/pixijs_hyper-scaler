# ドキュメント・資料の配置ガイド（ゲーム制作向け）

プログラミング以外の資料（ライブラリ公式資料、作品設定、世界観など）を整理するための推奨フォルダ構造です。

## 推奨フォルダ構造

```
プロジェクトルート/
├── docs/                    # ドキュメント全体のルート
│   ├── reference/           # 参照用・公式資料
│   │   ├── engine/          # ゲームエンジン公式ドキュメント（コピーやリンク集）
│   │   ├── libraries/       # 使用ライブラリの公式資料・APIリファレンス
│   │   ├── platforms/       # プラットフォーム公式ドキュメント（例: mixi2 developer platform）
│   │   └── tools/           # ツール（エディタ・アセットツール等）のマニュアル
│   │
│   ├── design/              # 作品設計・設定
│   │   ├── world/           # 世界観・設定・用語集
│   │   ├── lore/            # ロア（神話・歴史・伝承・世界内文献）
│   │   ├── characters/      # キャラクター設定・台本
│   │   ├── story/           # シナリオ・ストーリー構成
│   │   ├── mechanics/       # ゲームシステム設計（仕様書）
│   │   └── levels/          # ステージ・マップ設計
│   │
│   ├── specs/               # 仕様書・企画書（tech=実装方針, gdd=ゲーム仕様で使い分け）
│   │   ├── tech/            # 実装方針（レート制限の回避、gRPC再接続など）
│   │   ├── gdd/             # ゲーム仕様（謎解きの正解判定、ボードゲームの勝敗条件など）
│   │   └── milestones/      # マイルストーン・タスク一覧
│   │
│   └── assets-doc/          # アセットに関するメモ（ライセンス・出典など）
│       ├── art/
│       ├── audio/
│       └── fonts/
```

## 各フォルダの役割

| フォルダ | 用途 | 例 |
|----------|------|-----|
| **reference/** | 外部の公式資料。コードに直接関係しない「参照用」の置き場。 | Unity/Unreal マニュアル、ライブラリ API ドキュメントのローカルコピーやリンク集 |
| **design/** | 作品固有の設定・世界観・ストーリー。プログラミングに依存しない創作資料。 | 用語辞典、キャラ表、シナリオ本、**ロア**（神話・歴史・伝承）、ステージ構成メモ |
| **specs/** | 企画・仕様の「文書」。**tech/**＝実装方針、**gdd/**＝ゲーム仕様で分けると AI への指示がスムーズ。 | tech: レート制限回避方針、gRPC再接続。gdd: 謎解き正解判定、ボードゲーム勝敗条件。milestones: リリース計画 |
| **assets-doc/** | 画像・音声・フォントなどの出典・ライセンス・利用条件のメモ。 | 使用 BGM 一覧、フォントライセンス表 |

## 運用のコツ

1. **reference/** には「リンク集」や「よく使うページのメモ」を置き、巨大な公式ドキュメントは URL で参照するのがおすすめです。
2. **GitHub のリポジトリをコーディングの参考にしたい場合**は、`reference/libraries/` などにクローンしておくと Cursor が中身を参照しやすくなります。手順は `docs/reference/README-github-reference.md` を参照。
3. **design/** は Markdown（.md）で書くとバージョン管理しやすく、Cursor でも検索しやすいです。
4. **specs/** は **tech/**（実装方針）と **gdd/**（ゲーム仕様）に分けると、謎解き・ボードゲームなどでもプロンプトが整理しやすいです。GDD などは PDF でも可。要約や目次を `specs/README.md` に書いておくと探しやすいです。
5. コード用の設定（例: `package.json`, `project.godot`）は従来どおりプロジェクトルートや `src/` などに置き、**docs/** には「説明・仕様」だけを置くようにすると分かりやすいです。

## このリポジトリでの使い方

- 上記の `reference/`, `design/`, `specs/`, `assets-doc/` は **docs/** の直下に作成して使います。
- 既存のゲームプロジェクトに **docs/** をそのままコピーして、必要なサブフォルダだけ使っても構いません。
